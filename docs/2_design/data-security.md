## 一、安全架构概述

### 1.1 安全设计原则

| 原则 | 说明 |
|------|------|
| **数据本地化** | 所有数据存储在本地，无云端同步 |
| **加密存储** | 数据库文件整体加密，导出文件可选加密 |
| **应用绑定** | 加密导出文件仅本应用可解密 |
| **用户可选密码** | 支持用户设置密码增强安全性 |
| **透明加密** | 应用运行时数据明文，便于检索 |

### 1.2 安全边界

```
┌─────────────────────────────────────────────────────────────┐
│                        安全边界                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  应用启动 → 密码/指纹验证 → 解密数据库                       │
│                                                             │
│  数据存储 → SQLCipher加密 → 数据库文件加密                   │
│                                                             │
│  数据导出 → AES-256-GCM加密 → 仅本应用可解密                 │
│                                                             │
│  数据访问 → 数据库解锁后明文 → 无额外验证                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、数据库加密方案

### 2.1 加密架构

采用 **可选密码 + DEK+KEK 信封加密**：

```
用户选择是否设置密码
    │
    ├── 是 → 用户密码 → PBKDF2-HMAC-SHA256 (100000次迭代)
    │              │
    │              ▼
    │         ┌─────────────────┐
    │         │      KEK        │ ← 密钥加密密钥（存于Keystore）
    │         └────────┬────────┘
    │                  │ AES-256-GCM
    │                  ▼
    └──────────→ ┌─────────────────┐
                 │      DEK        │ ← 数据加密密钥（加密后存DB）
                 │  (无密码时直接存储)│
                 └────────┬────────┘
                          │ SQLCipher
                          ▼
                 ┌─────────────────┐
                 │  SQLite数据库   │ ← 整个数据库文件加密
                 │   (存储明文)    │
                 └─────────────────┘
```

### 2.2 密钥派生参数

| 参数 | 值 | 说明 |
|------|------|------|
| **KEK派生算法** | PBKDF2-HMAC-SHA256 | 密码派生 |
| **迭代次数** | 100,000 | 防暴力破解 |
| **盐值长度** | 16字节 | 随机生成 |
| **KEK长度** | 256位 | AES-256密钥 |
| **DEK长度** | 256位 | AES-256密钥 |

### 2.3 SQLCipher配置

| 配置项 | 值 |
|--------|------|
| **加密算法** | AES-256-CBC |
| **密钥派生** | PBKDF2-HMAC-SHA1 (SQLCipher内置) |
| **迭代次数** | 64,000 |
| **页面大小** | 4096字节 |

## 三、导出文件加密方案

### 3.1 加密算法

| 项目 | 值 |
|------|------|
| **加密算法** | AES-256-GCM |
| **IV长度** | 12字节 |
| **认证标签** | 16字节 (128位) |
| **压缩算法** | zlib |

### 3.2 密钥派生策略

#### 密码模式（KDF=0x01）

```
KDF: PBKDF2-HMAC-SHA256
输入: 用户密码(UTF-8) + 应用签名SHA256拼接
盐: 随机16字节（存文件头）
迭代: 100,000次
输出: 32字节AES密钥

安全性：需同时拥有用户密码和应用签名才能解密
```

#### 无密码模式（KDF=0x02）

```
KDF: HKDF (RFC 5869)
固定盐: Hermes_Export_Fixed_Salt_v1
输入密钥材料: 应用签名SHA256
信息标识: account_export
输出: 32字节固定密钥

安全性：相同签名应用可自动解密，可被逆向工程破解
```

### 3.3 安全性评估

| 模式 | 安全级别 | 风险说明 | 适用场景 |
|------|----------|----------|----------|
| 密码模式 | ⭐⭐⭐⭐⭐ | 需密码+签名才能解密 | 重要数据备份、跨设备迁移 |
| 无密码模式 | ⭐⭐⭐☆☆ | 可被逆向工程破解 | 临时备份、用户对安全要求不高 |

### 3.4 应用签名获取

**Android实现**：

```kotlin
fun getAppSignatureSHA256(context: Context): ByteArray {
    val packageInfo = context.packageManager
        .getPackageInfo(context.packageName, PackageManager.GET_SIGNATURES)
    val signature = packageInfo.signatures[0]
    val md = MessageDigest.getInstance("SHA-256")
    return md.digest(signature.toByteArray())
}
```

**iOS实现**（建议方案）：

使用Bundle ID + Team ID组合作为签名标识，或通过SecCodeCopySigningInformation获取签名信息。

## 四、密钥管理

### 4.1 密钥存储位置

| 密钥类型 | 存储位置 | 说明 |
|----------|----------|------|
| **KEK** | Android Keystore | 不可导出，硬件保护 |
| **加密后的DEK** | 数据库 key_storage表 | KEK加密后存储 |
| **导出密钥** | 运行时派生 | 不存储，每次导出重新派生 |

### 4.2 密钥生命周期

```
数据库密钥：创建 → 使用 → 更新（密码修改） → 销毁（数据清空）
导出密钥：派生 → 加密数据 → 销毁（导出完成）
导入密钥：派生 → 解密数据 → 销毁（导入完成）
```

### 4.3 密钥更新流程

```
用户修改密码：
    → 输入当前密码（验证）
    → 输入新密码
    → 用当前KEK解密DEK
    → 用新密码派生新KEK
    → 用新KEK加密DEK
    → 更新key_storage表
```

## 五、安全警告规范

### 5.1 必须显示安全警告的场景

| 场景 | 警告内容 |
|------|----------|
| **导出明文JSON** | 导出为明文JSON文件，数据无加密保护。任何人均可查看文件内容，存在隐私泄露风险。 |
| **导出不设密码** | ⚠️ 此文件无密码保护，任何拥有本应用的人都能打开。请勿分享或上传到云端。 |
| **关闭应用密码** | 关闭密码保护后，任何人都能打开本应用查看数据。 |
| **清空所有数据** | 此操作将删除所有验证渠道、账号和绑定关系数据，且不可恢复。 |
| **忘记密码** | 重置密码将清空所有数据，您将丢失所有已保存的账号信息。 |

### 5.2 警告文案规范

```
警告文案格式：
- 使用⚠️图标引起注意
- 清晰描述风险后果
- 提供替代建议（如"建议使用安全导出"）
- 需用户勾选确认"我已了解风险"
```

### 5.3 防呆设计规范

```
防呆设计要求：
- 删除账户：需输入账户名称确认
- 清空数据：需输入"CLEAR"确认
- 危险操作：使用红色警告色按钮
- 不可逆操作：明确标注"此操作不可撤销"
```
