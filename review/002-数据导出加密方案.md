## 数据导出加密技术方案

背景现状：

- **数据存储现状**：应用内 SQLite + SQLCipher 对数据库文件整体加密，但应用运行时数据是明文的（透明加密）。
- **导出目标**：用户可将数据导出为加密文件，**可选择设置密码**；导出的文件**仅本应用（相同签名）可以打开**，其他应用无法解析或解密。
- **跨设备迁移**：用户可以在新设备上安装本应用，导入之前导出的文件。若文件设置了密码，则需输入密码；若未设置密码，则仍能被本应用打开（无需密码），但其他应用不行。

因此，Hermes导出的数据：**只有签名相同的应用才能成功解密导入**，不依赖特定硬件或设备。用户不设置密码时，也要保证这一点，且支持跨设备。

---

### 1. 设计目标
- **加密导出**：数据强加密（AES-256-GCM）。
- **用户可选密码**：支持用户设置密码；若未设置，导出文件仍然加密，但使用应用级固定密钥。
- **应用绑定**：无论是否设置密码，解密时都需要应用签名匹配（或内置固定密钥）。其他应用无法解密。
- **跨设备迁移**：同一应用（相同签名）在新设备上可正常导入（无密码模式无需用户输入，密码模式需输入正确密码）。

### 2. 总体架构
采用 **自定义二进制容器 + AES-256-GCM**。不同之处在于密钥派生策略：

| 模式               | 密钥来源                                                     | 跨设备支持     | 是否需要用户输入   |
| ------------------ | ------------------------------------------------------------ | -------------- | ------------------ |
| **用户设置密码**   | PBKDF2(用户密码 + 应用签名 + 随机盐)                         | 是（签名相同） | 是（导入时需输入） |
| **用户不设置密码** | 固定密钥，从应用签名通过 HKDF 派生（或编译时生成固定 AES 密钥，并与签名绑定校验） | 是（签名相同） | 否                 |

### 3. 密钥管理详细设计

#### 3.1 用户设置密码模式
- **KDF**：PBKDF2-HMAC-SHA256  
  - 输入：用户密码（UTF-8） + 应用签名 SHA256（二进制）拼接作为密码材料（`password + signature`）  
    
    > 注意：直接将签名混入盐值输入，可确保不同签名的应用即使知道用户密码也无法派生相同密钥。
  - 盐：随机 16 字节，存储于文件头。
  - 迭代次数：100,000
  - 输出：32 字节 AES-256 密钥
- **安全性**：只有同时知道用户密码且应用签名匹配才能解密，其他应用无法打开。

#### 3.2 用户不设置密码模式
- **设计思路**：不需要用户输入任何秘密，但必须保证只有本应用（相同签名）能解密。  
  因此可采用 **应用签名派生固定密钥**。
- **具体实现**：
  - 在应用代码中定义一个固定的“盐”字符串（例如 `"com.your.app.export.fixed.salt"`），与运行时获取的应用签名 SHA256 一起通过 HKDF 扩展出 32 字节密钥。
  - 或者更简单：直接使用应用签名的 SHA256 哈希值的前 32 字节作为 AES 密钥（但签名哈希可能不是均匀随机，推荐 HKDF）。
  - 派生公式：  
    `fixedKey = HKDF-Extract(salt = 固定字符串, IKM = appSignatureSHA256)`  
    `fixedKey = HKDF-Expand(fixedKey, info = "account_export", L = 32)`
  - 该密钥在应用每次运行时均可重新计算，不依赖 Keystore 硬件，因此跨设备完全一致（只要应用签名相同）。
- **导出加密**：直接使用 `fixedKey` + 随机 IV 进行 AES-256-GCM 加密，不存储盐，KDF 算法 ID 标记为 `0x02`（固定密钥模式）。
- **导入解密**：相同应用重新计算 `fixedKey`，读取 IV 和密文，尝试解密。成功则数据有效；失败（例如文件被篡改）则报错。
- **安全性评估**：
  - 优势：任何拥有相同签名的应用都能打开（跨设备），用户无需记忆密码，便利。
  - 风险：如果攻击者逆向 APK 获取到固定盐和派生算法，并获取到应用签名（可从 APK 中提取），则理论上可以计算 `fixedKey`，从而解密所有未设置密码的导出文件。**但前提是攻击者必须能够拿到你的导出文件**。这属于“无密码”模式的固有问题，需要在 UI 中明确警告用户：**未设置密码的导出文件安全性较低，请勿分享给他人或存储在不安全位置**。
  - 防护增强：可以在派生时引入一个仅在编译时随机生成的硬编码种子（混淆存储），增加逆向难度，但无法绝对防止。考虑到用户主动选择“不设置密码”，可认为其接受较低安全级别。

#### 4. 导出文件格式（更新）

| 偏移     | 字段       | 大小   | 说明                                                         |
| -------- | ---------- | ------ | ------------------------------------------------------------ |
| 0        | 魔数       | 4字节  | `0xAAFF0101`                                                 |
| 4        | 版本号     | 1字节  | `0x01`                                                       |
| 5        | 加密算法ID | 1字节  | `0x01` = AES-256-GCM                                         |
| 6        | KDF算法ID  | 1字节  | `0x01` = PBKDF2（用户密码模式）<br>`0x02` = 应用固定密钥模式（无用户密码） |
| 7        | 标志位     | 1字节  | bit0: 压缩启用（zlib）                                       |
| 8        | 盐长度     | 1字节  | 若 KDF=0x01 则为16，若 KDF=0x02 则为0                        |
| 9        | 盐         | 变长   | 仅 KDF=0x01 时存在                                           |
| 9+盐长   | IV长度     | 1字节  | 固定12                                                       |
| 9+盐长+1 | IV         | 12字节 | 随机生成                                                     |
| 后       | 认证标签   | 16字节 | GCM Tag                                                      |
| 后       | 密文长度   | 4字节  | 大端序                                                       |
| 后       | 密文       | 变长   | 加密后的压缩数据                                             |

> 对于 KDF=0x02（无密码模式），文件头中不存储盐，盐长度字段为0。

### 5. 数据序列化与压缩
- 从 SQLCipher 解密后的数据库中读出所有业务表数据，转为 JSON。
- 敏感字段（如密码）在导出时保持原始明文（因为数据库内也是明文存储）。  
  注意：如果未来云端存储需要加密，可在导出前再次加密敏感字段，但当前方案直接导出明文（因为导出文件本身已加密）。
- 使用 zlib 压缩 JSON 字符串。

### 6. 核心流程实现要点

#### 6.1 导出流程伪代码（Android）
```kotlin
fun exportData(context: Context, password: String?, outputStream: OutputStream) {
    // 1. 从数据库读取所有数据 -> JSON 明文
    val jsonData = exportAllTablesToJson()   // 已经是明文
    val jsonBytes = jsonData.toByteArray(Charsets.UTF_8)
    val compressed = zlibCompress(jsonBytes)
    
    // 2. 生成随机 IV
    val iv = ByteArray(12).also { SecureRandom().nextBytes(it) }
    
    // 3. 派生密钥
    val secretKey = if (password != null) {
        // 密码模式：PBKDF2(用户密码 + 应用签名)
        val appSignature = getAppSignatureSHA256(context)
        val passwordMaterial = (password + appSignature.toHexString()).toByteArray()
        pbkdf2(passwordMaterial, salt = random16Bytes, iterations = 100000, keyLen = 32)
    } else {
        // 无密码模式：应用固定密钥
        computeFixedKeyFromAppSignature(context)
    }
    
    // 4. AES-GCM 加密
    val cipher = Cipher.getInstance("AES/GCM/NoPadding")
    cipher.init(Cipher.ENCRYPT_MODE, SecretKeySpec(secretKey, "AES"), GCMParameterSpec(128, iv))
    val ciphertextWithTag = cipher.doFinal(compressed)
    // ciphertextWithTag 最后16字节是 tag，前面是密文
    val tag = ciphertextWithTag.copyOfRange(ciphertextWithTag.size - 16, ciphertextWithTag.size)
    val encryptedData = ciphertextWithTag.copyOfRange(0, ciphertextWithTag.size - 16)
    
    // 5. 写入文件头（根据是否密码模式写入不同字段）
    writeExportFile(outputStream, password != null, salt, iv, tag, encryptedData)
}
```

#### 6.2 无密码模式的固定密钥派生（Android 示例）
```kotlin
fun computeFixedKeyFromAppSignature(context: Context): ByteArray {
    val appSignature = getAppSignatureSHA256(context)   // 32字节
    val fixedSalt = "YourAppName_Export_Fixed_Salt_v1".toByteArray()
    // 使用 HKDF (RFC 5869)
    val hkdf = HKDF("HmacSHA256")
    val pseudoRandomKey = hkdf.extract(salt = fixedSalt, ikm = appSignature)
    val okm = hkdf.expand(pseudoRandomKey, "account_export".toByteArray(), 32)
    return okm
}
```
> 注意：`getAppSignatureSHA256` 应返回当前应用签名（APK 签名）的 SHA256 哈希。同一开发者签名在不同设备上相同。

### 7. 平台实现差异
- **iOS**：获取应用签名可通过 `SecCodeCopySigningInformation` 获取 TeamIdentifier 或证书信息，但推荐使用 Bundle ID + 固定的 App Group 标识符组合（因为 iOS 应用签名机制不同）。更简单的方式：在 iOS 上使用 Keychain 存储一个随机生成且不可导出的密钥，但这样无法跨设备。为了跨设备，iOS 也应采用类似 Android 的固定密钥派生方法，从应用签名（例如通过 `kSecAttrApplicationTag` 获取）派生。具体可参考使用 `SecKeyCopyExternalRepresentation` 获取公钥哈希。

### 8. 安全性补充说明
| 模式           | 优点                                                         | 风险                                       | 适用场景                                 |
| -------------- | ------------------------------------------------------------ | ------------------------------------------ | ---------------------------------------- |
| 用户设置密码   | 强安全，即使导出文件泄露，攻击者没有密码且无法猜到（签名组合） | 用户需记住密码                             | 跨设备迁移、重要数据备份                 |
| 用户不设置密码 | 便捷，无需密码即可跨设备恢复                                 | 应用固定密钥可能被逆向工程，导致文件被解密 | 临时备份、用户对安全要求不高且仅自己使用 |

**强烈建议**：在 UI 上明确警告无密码模式的风险，并提供默认建议使用密码。同时，对于无密码模式导出的文件，可在文件头中嵌入应用签名哈希的校验值，防止文件被恶意修改后替换为其他数据。

### 9. 用户交互流程
- **导出时**：
  - 选项一：设置密码（输入框，需确认密码）→ 加密导出。
  - 选项二：不设置密码 → 弹出警告：“导出文件将被加密但仍可被任何拥有本应用相同签名的人打开（如果攻击者逆向获得固定密钥）。建议仅在可信环境使用。是否继续？”
- **导入时**：
  - 如果检测到文件是密码模式，弹出密码输入框。
  - 如果检测到无密码模式，直接尝试解密，失败则提示“文件损坏或不是本应用导出的数据”。

### 10. 总结
- ✅ 用户可选择是否设定密码。
- ✅ 导出文件仅本应用（相同签名）可打开，其他应用无法解密。
- ✅ 支持跨设备迁移（同一签名应用在新设备上导入时，密码模式需输入密码，无密码模式无需输入）。
- ✅ 数据在应用内明文存储（SQLCipher 透明加密），导出时直接获取明文后加密。

实现时请注意派生固定密钥的保密性（可增加代码混淆），并为无密码模式提供明确的安全提示。
